## 1. 输入约定（喂给 LLM 的 user-message）

转录文本在 LLM 调用前由 main 进程组装。**所有模板共享同一份输入格式**，模板只决定 system / instruction，不决定数据 schema。

### 1.1 文本组装规则

```
[mic 00:00:12] 我们今天主要聊一下下季度的目标
[system 00:00:18] 好的，那我先说一下我这边的看法
[mic 00:00:32] 嗯，你说
[system 00:00:35] 我觉得 Q2 应该把重点放在留存上...
```

- **每段一行**，格式 `[<speaker> <HH:MM:SS>] <text>`
- `speaker` 取值 `mic` / `system` —— 来源于 LazyAudio 双轨录音的物理音源，不是说话人分离（spike-005 / audio-capture.md）；模板里 `mic = 我`，`system = 对方 / 会议内其他人`
- 时间戳 `HH:MM:SS` 来源于段落的 `startMs`，向下取整到秒
- 段落顺序 = 时间顺序（不是 mic 全部 → system 全部）
- hypothesis 不进 LLM；只有 confirmed 段落进
- mic 路若关（用户只录系统音），所有段都是 `[system ...]`；system 路若关同理

### 1.2 长度处理

| 转录长度（粗估 token） | 处理                                                                                                               |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------ |
| < 80% 上下文窗         | 整篇一次塞进去                                                                                                     |
| 80% – 100%             | 模板里 `## 来源` 段保留首尾各 20%，中间用 `[...截断 N 段，约 M 分钟...]` 占位                                      |
| > 100%                 | v0.1 弹错："录音过长，超出当前模型上下文。请切换更大上下文的模型或导出转录文本手动总结。" 不做 map-reduce（v0.2+） |

token 估算用 `text.length / 1.5`（中文粗估）；上下文窗按用户在设置里选的 model 取，无法识别则按 8k 保守值。

### 1.3 缺失内容如何告诉模型

转录里 mic 或 system 路无内容时，组装层在文本开头插入 sysmeta 行：

```
[sysmeta] 本次录音仅 mic 路有内容（用户关闭了 system 音源）
```

模板里要在 prompt 顶部告诉 LLM："如遇 `[sysmeta]` 开头的行，是 LazyAudio 提供的元数据，不是录音内容；请据此调整输出。"

## 2. 模板：会议纪要 (meeting)

### 2.1 元数据

| Field            | Value                                 |
| ---------------- | ------------------------------------- |
| name             | 会议纪要                              |
| icon             | `users`                               |
| 应用于           | `meeting`（默认）；可手选给其他类型用 |
| 输出格式         | Markdown                              |
| 默认 max_tokens  | 2048                                  |
| 默认 temperature | 0.3（结构化任务，低温度）             |

### 2.2 System prompt

```
你是一个专业的会议纪要助手。下面 user 消息里是 LazyAudio 录制的一段会议的转录文本。

约定：
- 每段格式 `[<speaker> <HH:MM:SS>] <text>`
- speaker 是 `mic`（= 我，记录者本人）或 `system`（= 对方 / 会议内其他人）
- 段落按时间顺序排列；只包含确认稿（confirmed），不含 hypothesis
- 如遇 `[sysmeta] ...` 开头的行，是 LazyAudio 提供的元数据（如"仅 mic 路有内容"），据此调整输出，不要把它当成会议内容

请生成一份结构化的会议纪要，结构如下：

## 议题
列出本次会议讨论的主要议题（3-7 条）。每条一句话点题。

## 决议
明确达成的决议。每条 `- {决议内容}（{责任人}）`。若责任人未明确，写"（未指定）"。

## 待办
具体的 action items，格式：`- [ ] {内容} @{责任人} {期限}`。如果未明确，用 @未指定 / 期限未指定。

## 悬而未决
没有结论但需要后续跟进的议题。每条一句话 + 建议下一步。

## 摘要
用 2-3 句话总结本次会议核心。

要求：
- 用中文输出
- 不要编造转录里没有的内容；只用转录里出现过的事实
- 引用具体讨论时用时间戳 `[HH:MM:SS]` 锚定（便于用户在 LazyAudio 里跳回去听）
- 如果转录不清晰、不完整、或某节没有讨论到，对应小节明确写"（本次无）"，不要硬凑
```

### 2.3 User prompt

仅转录文本（§1.1 格式）。无额外 wrapper。

### 2.4 边界

- 转录 < 30s：LLM 大概率啥都总结不出。**模板不变，仍调用**；输出会很短或全是"（本次无）"，让用户自己看到。
- 仅 system 路有内容（用户没录 mic）：sysmeta 提示后，LLM 仍按结构出（"我"视角的内容会缺，会议纪要仍可用）

## 3. 模板：要点速记 (note)

### 3.1 元数据

| Field            | Value                                               |
| ---------------- | --------------------------------------------------- |
| name             | 要点速记                                            |
| icon             | `pencil`                                            |
| 应用于           | `note`（默认）+ `通用` fallback；可手选给其他类型用 |
| 输出格式         | Markdown                                            |
| 默认 max_tokens  | 1536                                                |
| 默认 temperature | 0.4                                                 |

### 3.2 设计意图

note 场景 = 用户对着电脑自言自语，把脑子里的散乱想法念出来。**没有"对方"**，输入几乎全是 `[mic ...]`，所有内容都是用户自己说的。

note 不要"议题 / 决议 / 待办"这种会议结构，要"把散乱思路归到几条清晰要点上 + 抓出隐藏的 action"。结构借鉴 [Tiago Forte 的 CODE / Building a Second Brain] 的精简版：先压缩成主旨，再分类聚合到要点，最后单独捞 action / 待思考。

### 3.3 System prompt

```
你是一个个人笔记整理助手。下面 user 消息里是用户对着电脑自言自语录下的一段散乱笔记的转录文本。

约定：
- 每段格式 `[<speaker> <HH:MM:SS>] <text>`
- 这个场景下 speaker 大多是 `mic`（= 用户本人）；偶尔有 `system` 的内容是用户录音时电脑播放的背景内容，**可作为上下文参考但不要当成用户的观点**
- 如遇 `[sysmeta] ...` 开头的行，是 LazyAudio 提供的元数据，不是笔记内容

用户的目标是把刚才说出来的散乱想法整理成可以快速回看的笔记。请生成如下结构：

## 一句话
1 句话概括用户这段笔记的核心。

## 要点
把内容归到 3-7 条要点，每条：
- 一个 **加粗短语** 点题
- 1-3 句展开
- 若该要点来自具体某段，末尾加 `[HH:MM:SS]` 锚定

要点之间是**主题平行关系**而不是时间顺序——把同一主题的散落片段合并到同一条；用户东扯西扯本来就是常态。

## 待办 / 待思考
- `- [ ] {action 内容}` —— 用户说出来明确要做的事
- `- ? {待思考问题}` —— 用户提了但没结论、值得后续想清楚的问题

若没有，写"（本次无）"，不要硬凑。

## 原话引用
最多 3 条用户说得**特别精炼或值得保留的原话**，格式 `> {原文} —— [HH:MM:SS]`。
判断标准：用户自己听到这句话会觉得"对，我就是这个意思"。
若全篇都很口语化没有金句，写"（本次无）"。

要求：
- 用中文输出
- 忠实于用户说过的内容，不编造、不引申、不提出用户没提过的方案
- 口语化的 "嗯 / 那个 / 就是 / 然后" 等填充词要清洗掉
- 重复说的同一观点合并；用户改主意了取最后一次的说法（前后矛盾时在该要点末尾加 `[前后有改口]`）
```

### 3.4 User prompt

仅转录文本（§1.1 格式）。

### 3.5 边界

- system 路有大量内容（用户录笔记时电脑在播视频）：sysmeta 提示后，模板已说明把 system 当背景；实际输出仍会偶尔混入背景内容。**v0.1 接受这个噪声**，v0.2 考虑录前 prompt 用户"建议关闭背景声"
- 录音 < 15s：LLM 输出会很瘪，**模板不变**。

## 4. 输出渲染约定

LLM 返回的 Markdown 直接渲染在主窗口详情区的"摘要"tab（[`information-architecture.md` §3.2](./information-architecture.md)）。

| Markdown 元素           | 渲染行为                                                                                                         |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `## 标题`               | 24px gray-900 weight 600，section 间距 20                                                                        |
| `- [ ] / - [x]` 待办    | 渲染成可点击 checkbox；勾上 = 写回 meta `summary.todos[].done`（v0.1 不同步回 markdown 文件，状态在 meta 里）    |
| `[HH:MM:SS]` 时间戳锚点 | 可点击；点击 = 主窗口播放器 seek 到该位置（[`information-architecture.md` §7.5](./information-architecture.md)） |
| `> 引用`                | 左 4px accent border + gray-50 背景；mono 不变                                                                   |
| **粗体** / _斜体_       | 默认 markdown 渲染                                                                                               |
| 代码块                  | mono 字体 + gray-50 背景；v0.1 不做语法高亮（摘要里出现代码块的概率低）                                          |

**不渲染**的内容：raw HTML、image（LLM 不该返回这些；返回了直接 escape）。

## 5. 自定义模板

用户在 [`screen-specs/settings.md` Tab 4](./screen-specs/settings.md#tab-4--llm-模板) 里可：

1. 改预置模板的 prompt 文本（重置回默认 = 删除 user override，回退到本文档定义的版本）
2. 新建模板（fork 一个预置模板编辑）
3. 改"应用于会话类型" multi-chip：决定录音 sessionType → 自动选哪个模板

v0.1 **不**给：模板变量编辑器、prompt 模板语法（Jinja / Handlebars）、多 LLM provider 不同 prompt。
